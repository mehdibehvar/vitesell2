Four React and Next.js technical interview challenges

Medium React hooks Closures State
Fix the stale closure in a counter component
Identify why the interval reads wrong values and correct it without breaking the UI
15 min
Buggy component
import { useState, useEffect } from 'react'; function AutoCounter() { const [count, setCount] = useState(0); useEffect(() => { const id = setInterval(() => { setCount(count + 1); // always logs 1, never increments }, 1000); return () => clearInterval(id); }, []); return <p>Count: {count}</p>; }
Observations from the team
  • The counter displays 1, 1, 1… instead of 1, 2, 3…
  • Adding count to the deps array "fixes" it but creates a new interval every second
  • The same pattern breaks in a form that auto-saves every 30 s
Fix the stale closure. Then explain the two valid approaches and the trade-off between them. Be ready to discuss when each is preferred.
Stuck?
The closure captures count at the time the effect ran. One fix: use the functional updater form setCount(c => c + 1). The other: use a useRef to hold a mutable reference to the latest count.
Medium Performance Memoization useCallback
Stop a child component from re-rendering on every keystroke
A parent's search input causes an expensive list to re-render 100+ times per second
20 min
Current code
// Parent.jsx function SearchPage() { const [query, setQuery] = useState(''); const [results, setResults] = useState(expensiveData); const handleDelete = (id) => { setResults(prev => prev.filter(r => r.id !== id)); }; return ( <> <input value={query} onChange={e => setQuery(e.target.value)} /> <ResultList items={results} onDelete={handleDelete} /> </> ); } // ResultList.jsx — wrapped in React.memo const ResultList = React.memo(({ items, onDelete }) => { // renders 500 rows — expensive return items.map(i => <Row key={i.id} item={i} onDelete={onDelete} />); });
Problem observed
  • React.memo is on ResultList but it still re-renders on every keystroke
  • Profiler shows 60–80 ms render blocking the main thread
  • The team added useCallback to handleDelete but it didn't help
Explain why React.memo isn't working despite being applied. Fix it correctly and describe when memoization actually hurts performance instead of helping.
Stuck?
React.memo does a shallow comparison of props. Every render creates a new handleDelete function reference — so the prop always looks "changed". Wrap it in useCallback with the correct dependency array. Then ask: what's in the deps array and why?
Hard Next.js App router RSC Data fetching
Architect a dashboard page using the App Router correctly
Decide which parts are Server Components, Client Components, or streamed — and justify every choice
25 min
Page layout to implement
Revenue chart — fetches from a slow analytics API (~2 s)
User profile header — uses cookies() to identify the logged-in user
Recent activity feed — fast DB query, no interactivity
Notification bell — client-side polling every 10 s
Search bar — updates URL params on submit
Refresh button — calls a Server Action to revalidate cache
Constraints
  • Must use Next.js 14 App Router — no getServerSideProps
  • Revenue chart must not block initial HTML delivery
  • No 'use client' on the root layout or any shared wrapper
  • Search state should survive a full page refresh
Label each component Server, Client, or Streamed via Suspense. Explain what breaks if you mark the wrong one 'use client' and how RSC reduces the client bundle.
Stuck?
Start by asking: does this component use useState, browser APIs, event handlers, or cookies()? If yes → Client. Then ask: is it slow? If yes → wrap it in <Suspense> with a skeleton fallback so it streams in after the shell is delivered.
Hard Custom hooks TypeScript Abstraction
Build a reusable useFetch hook with cancellation and caching
Three components in the same app are copy-pasting the same fetch + loading + error pattern
30 min
Pattern being duplicated everywhere
// Copied into UserCard, ProductGrid, OrderHistory… const [data, setData] = useState(null); const [loading, setLoading] = useState(true); const [error, setError] = useState(null); useEffect(() => { fetch(url) .then(res => res.json()) .then(setData) .catch(setError) .finally(() => setLoading(false)); }, [url]);
Your hook must handle
Abort in-flight requests when url changes or component unmounts
Simple in-memory cache — don't re-fetch the same URL twice
A refetch() function that bypasses the cache on demand
Fully typed with TypeScript generics: useFetch<T>(url)
Write the hook from scratch. Then explain how this compares to using a library like React Query or SWR — when would you roll your own vs. reach for a library?
Stuck?
Use AbortController inside useEffect and call controller.abort() in the cleanup function. Store the cache as a useRef holding a Map<string, T> so it persists across renders without triggering them.
